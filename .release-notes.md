## v0.1.7 — fix SSE proxy hang for remote-upstream MCPs

Followup to v0.1.6. The endpoint-rewriting proxy was wired in correctly but the response never reached the client: Fastify's `reply.send()` doesn't consume a Whatwg `ReadableStream` (which is what Node 20's global `fetch()` returns as `Response.body`). The SSE frames were buffering into the void, so any remote-upstream MCP opened through Pier just hung with no headers.

Fix: convert the web stream to a Node `Readable` via `Readable.fromWeb(...)` before handing it to Fastify, plus add `x-accel-buffering: no` on the proxied response to match what stdio-SSE already does.

Only affects `transport: http|sse` entries served via `/mcp/:name/sse` (the yantrikdb-through-Pier path). stdio-backed MCPs (where Pier spawns a subprocess and fronts its own SSE) were already writing through `reply.raw` and were unaffected.

CLI version bumped in lockstep so `pier` and the image stay on the same minor.

Upgrade with `npm i -g mcpier` or `docker pull ghcr.io/spranab/mcpier:latest`.
